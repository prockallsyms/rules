## Summary

In `github.com/jackc/pgx/v5/pgconn`, the `(*Pipeline).Sync()` method lacked the closed-pipeline guard that every other `Pipeline` method (`SendPrepare`, `SendDeallocate`, `SendQueryParams`, `SendQueryPrepared`, `Flush`) already carried. When `StartPipeline` is called on a `PgConn` that is busy or closed, `lock()` returns an error and the returned `*Pipeline` is initialized with `closed=true, err=<lock error>`. If the caller then invokes `Sync()` on that pipeline, the pre-fix code bypassed the closed check, called `p.conn.frontend.SendSync(...)` and then `Flush()` → `flushWithPotentialWriteReadDeadlock()` on a connection in an invalid/nil state, causing a panic. CWE-476 (Nil Pointer Dereference) / CWE-754 (Improper Check for Unusual Conditions). Impact: unauthenticated or application-level denial of service — any goroutine that calls `pipeline.Sync()` after receiving a closed-pipeline handle will crash the process.

## Detection discriminator

This engine has no library-version gate, so the rule proves the **vulnerable code structure** and is silent on the patched build.

The v5.5.2 fix adds `if p.closed { ...; return errors.New("pipeline closed") }` at the
top of `(*Pipeline).Sync`. `errors.New` is INLINED (errors.go:62) and lowers to an
allocation of an `errors.errorString`, i.e. a `CALL runtime.newobject` inside Sync.

Live disassembly (go tool objdump, unique CALLs in Sync):
- v5.5.1 (VULN):    {pgproto3.(*Frontend).SendSync, pgconn.(*Pipeline).Flush, runtime.morestack_noctxt}
- v5.5.2 (PATCHED): the above + `runtime.newobject`

So the engine-matchable handle is: inside the scoped function `pgconn.(*Pipeline).Sync`,
`context:calls("runtime.newobject")`:
- VULN    -> 0  (fix-signal absent  -> FIRE)
- PATCHED -> 1  (fix-signal present -> nil)

Confirmed live with a probe rule (`result:info`, description carries the count):
VULN newobject count=0 ; PATCHED newobject count=1.

## Reproducing the test binaries

```go
package main

import (
	"context"

	"github.com/jackc/pgx/v5/pgconn"
)

func main() {
	cfg, err := pgconn.ParseConfig("host=127.0.0.1 port=5432 user=nobody dbname=nodb")
	if err != nil {
		return
	}
	conn, err := pgconn.ConnectConfig(context.Background(), cfg)
	if err != nil {
		return
	}
	pipeline := conn.StartPipeline(context.Background())
	_ = pipeline.Sync()
	_ = pipeline.Close()
}
```

**What the fix added:** At the top of `(*Pipeline).Sync()`, a guard block was inserted:

```go
if p.closed {
    if p.err != nil {
        return p.err
    }
    return errors.New("pipeline closed")
}
```

This is an early-return error path that mirrors the identical guard already present in `(*Pipeline).Flush()`. Without it, calling `Sync()` on a pipeline whose `closed` field is `true` (set by `StartPipeline` when `lock()` fails) would unconditionally call `p.conn.frontend.SendSync(...)` and then enter `Flush()` → `flushWithPotentialWriteReadDeadlock()` on a connection in busy/closed state, causing a nil-pointer panic. The fix adds exactly 7 lines; no other file is touched.
